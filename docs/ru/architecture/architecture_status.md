# Текущее состояние архитектуры

## Текущая структура

Приложение использует пять основных слоёв:

```text
src/ui
src/app
src/logic
src/core
src/platform
```

Runtime-состояние централизовано в `src/app/runtime`, а каждый проект использует единственную структуру workspace, описанную ниже.

## Зависимости

```text
core      → core
logic     → logic, core
platform  → platform, core
ui        → ui
app       → app, ui, logic, core, platform
```

Статические циклы и нарушения направления зависимостей проверяются автоматически.

## Границы platform, logic и core

`src/platform` содержит общие и переиспользуемые адаптеры окружения. Он не является единственным местом для всех файловых, сетевых и процессных операций.

`src/logic` владеет прикладными сценариями, включая неотделимый от них ввод и вывод: копирование, удаление, архивирование, загрузку и запись данных проекта.

`src/core` владеет низкоуровневыми алгоритмами и связанными с ними системными операциями. Прямой запуск процессов остаётся в реализациях форматов, когда выделение дополнительного адаптера не уменьшает связанность.

Новую общую сеть, общий запуск процессов и постоянное хранилище следует по возможности оформлять через platform-адаптеры. Специфический ввод и вывод остаётся рядом со сценарием или алгоритмом, если дополнительный адаптер не создаёт переиспользуемую границу.

## Рабочая структура проекта

Поддерживается одна схема:

```text
Projects/<имя>/input
Projects/<имя>/unpack
Projects/<имя>/output
```

`ProjectManager` создаёт её в рабочем каталоге из `settings.path`. Настройки для выбора другой структуры нет.

## UI и application

UI-модули импортируют только UI-модули и не создают domain request.

Application-слой связывает зависимости через composition, создаёт запросы, запускает сценарии и возвращает результаты в UI.

Project Workspace и Plugin Manager получают явные host-объекты. Application-состояние не хранится в случайных атрибутах Tk-окон.

Общий pipeline показа окон строит дочерние окна скрытыми, применяет тему, завершает ограниченную отрисовку первого кадра и только затем показывает окно. Startup splash и главное окно используют тот же механизм reveal; это отражено в полной карте архитектуры.

## Logic

Logic содержит реальное поведение распаковки, упаковки, импорта, преобразования, плагинов и инструментов.

Широкий перехват исключений допускается только на проверенных границах стороннего plugin-кода и в транзакционной очистке, которая повторно выбрасывает исходную ошибку. Архитектурные тесты получают и проверяют эту границу по текущему дереву исходников.

## Типизированные границы

Protocol сохраняются только там, где зависимость имеет несколько реализаций, используется несколькими модулями, меняется на внешней границе либо защищает UI от передачи слишком широкого runtime-объекта.

`tests/architecture/test_protocol_budget.py` задаёт проверенный верхний предел. Источником истины является результат живого теста, а не число, скопированное в документацию.

## Документация

Текущая английская документация находится в `docs/en`, русская — в `docs/ru`. Материалы из `docs/archive` не входят в активный набор и не определяют текущее поведение.

## Сборка и проверки

Release workflow собирает пользовательский артефакт. Pytest, Architecture Guard, Ruff, Mypy и явные ручные наборы из документа [Тесты и скрипты](../development/tests_and_scripts.md) запускает разработчик.

Исходники для разработки, тесты, скрипты, документация и workflow-файлы не входят в пользовательский release-архив.

## Правило сопровождения

Дальнейшие изменения должны опираться на реальные образы, bundled executables, roundtrip-сценарии и воспроизведённые функциональные ошибки. При намеренном изменении production-поведения тесты и документация обновляются вслед за ним; production-код не меняется только ради устаревшего ожидания.
