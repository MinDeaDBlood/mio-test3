# Архитектура MIO Kitchen

Этот документ описывает фактическую архитектуру текущего кода.

## Слои

Проект разделён на пять слоёв.

1. `src/ui` содержит окна, виджеты и представление состояния.
2. `src/app` связывает пользовательские действия с конкретными сценариями.
3. `src/logic` содержит правила, последовательности операций и прикладной ввод и вывод, который является частью конкретного сценария.
4. `src/core` содержит низкоуровневые алгоритмы, работу с форматами и системные операции, тесно связанные с этими алгоритмами.
5. `src/platform` содержит общие и переиспользуемые адаптеры окружения, настройки, языки, Git, desktop shell, сеть, процессы и persistent storage там, где они образуют самостоятельную границу.

Направление статических зависимостей закреплено архитектурными тестами.

```text
core      → core
logic     → logic, core
platform  → platform, core
ui        → ui
app       → app, ui, logic, core, platform
```

## UI

`src/ui` отвечает за отображение и ввод.

UI может создавать окна, виджеты, presentation state, ключи локализации и callback для application слоя.

UI не должен читать рабочие файлы проекта, запускать внешние процессы, создавать domain request или импортировать `src/app`, `src/logic`, `src/core` и `src/platform`.

## Application

`src/app` отвечает за композицию и координацию.

Application слой получает данные из UI, создаёт запросы, выбирает сценарий, запускает фоновые задачи, управляет состоянием окна и передаёт результат обратно в UI.

Application слой не должен реализовывать распаковку образов, обработку файловых систем и бинарные алгоритмы. Прямой ввод и вывод допускается только в небольших application boundary, где он является частью жизненного цикла процесса, окна или runtime.

## Logic

`src/logic` содержит прикладные сценарии программы.

Здесь находятся выбор операции, проверка условий, маршрутизация файлов, последовательность шагов, обработка результатов внешних инструментов, операции над проектами, плагинами и образами.

`logic` может напрямую использовать `os`, `pathlib`, `shutil`, `zipfile`, `requests` и другие библиотеки ввода и вывода, если операция является частью конкретного прикладного сценария. Сам по себе импорт `pathlib` или чтение структуры образа не считается нарушением архитектуры.

Для нового кода изменчивые границы, такие как сеть, запуск внешних процессов и постоянное хранилище, по возможности выносятся в `platform`, когда они переиспользуются, меняются независимо от прикладного сценария или имеют собственный жизненный цикл.

Logic не знает о конкретных Tk окнах и не получает целое состояние интерфейса.

Рабочая структура проекта фиксирована.

```text
Projects/<имя>/input
Projects/<имя>/unpack
Projects/<имя>/output
```

`input` содержит исходные архивы и образы.

`unpack` содержит распакованные разделы и их рабочие файлы.

`output` содержит готовые результаты сборки.

Текущие менеджеры и workflow принимают только эту структуру проекта.

## Core

`src/core` содержит низкоуровневые алгоритмы Android форматов, sparse image, boot image, GPT, архивов, файловых систем и бинарных структур.

Часть низкоуровневого кода сама запускает внешние процессы. Это относится, в частности, к `process_runner.py`, `image2chunks.py`, `blockimgdiff.py` и `Magisk.py`. Такие вызовы сохранены там, где процесс является частью алгоритма формата и его перенос не даёт отдельной переиспользуемой границы.

Новые пользовательские сценарии остаются в `logic` и `app`. Новый общий механизм запуска процессов создаётся в `platform`, если он нужен нескольким функциям или должен подменяться независимо от алгоритма.

## Platform

`src/platform` содержит общие адаптеры внешней среды.

Примеры:

1. Хранилище настроек.
2. Поиск языковых файлов.
3. JSON repositories.
4. Git операции.
5. Перезапуск приложения.
6. Desktop shell.
7. Настройка журналов.
8. Переиспользуемые файловые и процессные адаптеры.

`platform` не является универсальным слоем портов и адаптеров и не владеет всем вводом и выводом проекта. Операция остаётся в `logic` или `core`, если она является неотделимой частью конкретного сценария или алгоритма и не используется как общий адаптер.

Граница определяется не названием библиотеки, а назначением кода. Общая сеть, общий запуск процессов и постоянное хранилище относятся к `platform`. Работа с конкретным архивом, образом или рабочим каталогом внутри одного сценария может оставаться в `logic` или `core`.

## Окна и фокус

Все дочерние окна создаются через `src/ui/common/windowing.py`.

Класс `Toplevel` определяет владельца и устанавливает связь `transient`, пока дочернее окно ещё скрыто. Затем он применяет текущее оформление, выполняет ограниченную отрисовку первого кадра, перемещает окно в финальную позицию и только после этого показывает его и передаёт фокус. В Windows alpha gate, временное размещение за пределами экрана и DWM cloak не позволяют неопубликованному кадру без темы попасть в desktop compositor и при этом не закрепляют окно поверх других приложений.

Скрытые служебные окна не показываются принудительно. Нативные файловые диалоги получают активное окно MIO Kitchen через параметр `parent`.

## Runtime данные

Рядом с программой используются следующие каталоги.

```text
bin/
config/
languages/
plugins/
temp/
templates/
logs/
Projects/
```

`config` содержит редактируемые настройки.

`languages` содержит динамически обнаруживаемые JSON переводы.

`plugins` содержит локальную базу магазина и установленные плагины.

`temp` содержит только временные данные.

`templates` содержит шаблоны OTA и другие runtime шаблоны.

`bin` содержит исполняемые инструменты и связанные с ними ресурсы.

## Protocol

Protocol используется только для реальной границы.

Он оправдан, когда существует несколько реализаций, внешняя зависимость меняется независимо, один контракт используется несколькими модулями или необходимо не допустить передачу целого окна и глобального состояния.

Для единственной известной реализации внутри одного feature используется конкретный тип.

Архитектурный тест ограничивает общее количество Protocol классов, чтобы типизация не превращалась в отдельный слой инфраструктуры.

## Обработка исключений

В `logic` конкретные ошибки обрабатываются конкретными типами исключений.

Широкий `except Exception` разрешён только на явной границе стороннего плагина или для транзакционной очистки с обязательным повторным выбросом исходной ошибки.

Это правило закреплено AST тестом.

В `app` общий перехват допускается на границах произвольных callback, фоновых worker, Tk событий и внешних расширений, где приложение обязано сохранить работоспособность интерфейса.

## Стабильность

Дальнейшие архитектурные изменения должны быть локальными и привязанными к конкретной функции или воспроизведённой ошибке. Общая структура слоёв не должна перерабатываться без практического требования.
