# Логи и диагностика

Этот документ описывает текущую систему журналирования по реальному коду проекта. Логи нужны не только при аварии запуска. Они позволяют восстановить последовательность действий пользователя, фоновых задач, файловых операций, внешних процессов и плагинов.

## Где находится журнал

При запуске приложение создаёт отдельный файл в каталоге:

```text
logs/
```

Имя содержит дату, время и идентификатор процесса. После перезапуска, например при смене языка, новый процесс создаёт свой файл. Поэтому для анализа перезапуска обычно нужны два соседних файла.

Файловый обработчик принимает сообщения уровня `DEBUG` и выше. В журнал попадают обычные сообщения, предупреждения, ошибки и полные traceback.

## Общие поля записи

Каждая структурированная запись содержит:

| Поле | Назначение |
|---|---|
| дата и время | Точный момент события |
| уровень | `DEBUG`, `INFO`, `WARNING`, `ERROR` или `CRITICAL` |
| PID | Идентификатор процесса |
| поток | Главный поток или имя фоновой задачи |
| logger | Модуль или подсистема |
| файл и строка | Место, которое создало запись |
| operation | Текущая операция |
| op_id | Идентификатор конкретного запуска операции |

`operation` и `op_id` позволяют связать сообщения разных уровней в один сценарий. Например, запуск упаковки разделов получает отдельный `op_id`, а все сообщения о выбранных разделах, внешних инструментах и результате остаются связаны с этой операцией.

## Какие события записываются

### Запуск и завершение программы

Журнал фиксирует загрузку модулей, создание runtime session, чтение языка, создание главного окна, заставку, мастер первого запуска, вход в Tk mainloop, перезапуск и завершение процесса.

Необработанные исключения главного потока, фоновых потоков и Tk callback записываются с traceback.

### Пользовательские окна

Общий механизм окон фиксируется через журнал исключений Tk. Критические пользовательские сценарии дополнительно пишут события открытия, подтверждения и отмены. Например, окно упаковки разделов записывает выбранные разделы, параметры формы, ошибку проверки и передачу задачи в фон.

### Фоновые задачи

`UiTaskRunner` записывает имя worker, начало, успешное завершение и ошибку. Длительные операции получают собственный `operation_context` с временем выполнения.

### Файловые операции Python

После настройки основного журнала устанавливается системный audit hook Python. Он наблюдает каталоги конфигурации, плагинов, временных данных и рабочую область пользователя.

На уровне `INFO` записываются:

1. Открытие файла для записи, добавления, создания или изменения.
2. Создание каталога.
3. Удаление файла или каталога.
4. Переименование и перемещение.
5. Копирование файла или дерева.
6. Изменение прав и метаданных.
7. Создание временного файла или каталога.
8. Упаковка и распаковка архивов через `shutil`.

Операции чтения, `listdir` и `scandir` записываются на уровне `DEBUG`. Каталог `logs` исключён из файлового аудита, иначе запись самого журнала создавала бы бесконечный поток событий.

Чтобы технические события не забивали журнал, повторные одинаковые чтения в коротком интервале объединяются. Чтение исполняемого файла `tool.exe`, обращения к системным устройствам Windows вроде `NUL` и попытки повторно создать уже существующий каталог не записываются. Изменение, копирование, перемещение, удаление и создание новых файлов по прежнему фиксируются полностью.

Audit hook видит операции, выполненные кодом Python. Он не может увидеть каждое внутреннее действие отдельного бинарника. Для внешнего инструмента журналируется команда, PID, stdout, общий код возврата и длительность. Этого достаточно, чтобы определить границу, на которой внешний инструмент начал или закончил работу, и сопоставить её с созданными файлами.

### Внешние процессы

Общий process runner записывает:

1. Команду без секретных значений.
2. Путь к каталогу инструментов.
3. PID процесса.
4. Строки stdout и stderr.
5. Код возврата.
6. Длительность.
7. Ошибку отсутствующего файла или ошибку операционной системы.
8. Ситуацию, когда процесс остался активен во время очистки.

### Плагины

Плагинная подсистема записывает:

1. Поиск установленных плагинов.
2. Чтение `info.json`.
3. Состояние манифеста.
4. Зависимости и отсутствующие зависимости.
5. Наличие внешней точки входа `main.py` или `main.sh`.
6. Сформированный logic план выполнения.
7. Регистрацию Python entry point.
8. Запуск Python или shell точки входа.
9. Имена переданных параметров без раскрытия их значений.
10. Код завершения shell процесса.
11. Установку, удаление, экспорт и создание заготовки.

`main.sh` здесь означает файл внешнего установленного плагина в `plugins/installed/<id>/main.sh`. Это не служебный скрипт внутри `src`.

### Упаковка разделов

Сценарий упаковки записывает:

1. Открытие окна параметров и выбранные разделы.
2. Отмену или подтверждение формы.
3. Проверенные параметры упаковки.
4. Постановку фоновой задачи.
5. Пути `work`, `input` и `output`.
6. Чтение `parts_info`.
7. Тип каждого раздела.
8. Начало и завершение каждого раздела.
9. Выбор файловой системы и конвертации.
10. Вызовы внешних упаковщиков.
11. Ошибку этапа и итоговый результат.

После нажатия кнопки «Собрать» окно параметров закрывается. Сборка продолжается в фоне. Сообщения остаются видны в области логов главного окна и одновременно записываются в файл.

Событие плагинов `before_pack` вызывается только после подтверждения формы кнопкой «Собрать» и успешной проверки параметров. Простое открытие окна или отмена не запускают это событие.

## Уровни журналирования

| Уровень | Когда используется |
|---|---|
| `DEBUG` | Подробное состояние, чтение, выбор пути и технические переходы |
| `INFO` | Начало и завершение операции, запись файла, запуск процесса, результат |
| `WARNING` | Допустимая проблема, неподдерживаемый вариант или отклонённая операция |
| `ERROR` | Операция завершилась ошибкой, но приложение может продолжить работу |
| `CRITICAL` | Необработанное исключение или невозможность продолжать работу процесса |

## Как собирать данные для отчёта об ошибке

1. Закройте старый экземпляр программы.
2. Запустите программу и повторите проблему один раз.
3. Запишите точное действие, например «Проект → Собрать super → нажать Старт».
4. Закройте программу обычным способом, если интерфейс отвечает.
5. Приложите последний файл из `logs`.
6. Если перед проблемой был автоматический перезапуск, приложите два последних файла.
7. Приложите скриншот интерфейса, если ошибка связана с окном или расположением элементов.

Не нужно копировать только последние две строки. Полный файл содержит operation, op_id и предыдущие состояния, без которых причина может быть не видна.

## Проверки системы логирования

Основные автоматические проверки:

```bash
python tests/unit/platform/test_filesystem_audit.py
python tests/unit/platform/test_crash_logging.py
python tests/functional/ui/test_window_foreground.py
```

Полный ручной набор:

```bash
python scripts/manual/manual_unit_contracts.py
```

GUI сценарии:

```bash
python scripts/manual/manual_gui_smoke.py
```

GitHub Actions не запускает эти проверки автоматически. Они выполняются только вручную.
